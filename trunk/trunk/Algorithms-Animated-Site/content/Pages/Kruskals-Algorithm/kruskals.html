Kruskal's Minimum Spanning Tree Algorithm

Suppose you need to supply electrical power to an area that is currently off the grid. There is a set of cities and towns that need power and a set of potential connections between them, say roads or proposed right-of-ways. Each for each potential connection there is a cost. You need to find the minimum cost network that connects all the town. This is the problem that faced an employee of a Moravian power company, Jindřich Saxel. He posed the problem to a young mathematician, Otakar Borůvka, who, in 1926, published the problem and an algorithm to solve it. The problem turns out to have lots of applications besides electrical power distribution.

Modelling the situation as a graph problem, we have a set V of nodes (towns) and a set E of undirected edges (potential connections), each with an associated positive cost. We'll assume, for now, that the graph G = (V,E) is connected, i.e., there is at least one path between any two nodes. We need to find a subset T of the edges that connects all the towns and that has minimal total cost. A moment's thought will convince you that any solution T will form an undirected tree, since, if there were a cycle in T, we could remove one edge of the cycle to get a better solution. Since every node is in the tree, we call it a spanning tree and, as we want the one with least cost, we want a minimum spanning tree.

Idea

In 1956, U.S.A. mathematician Joseph Kruskal, having simplified Borůvka's algorithm, published a particularly simple method to find minimal spanning trees.

T := ∅

for each edge e in order of increasing cost

if T ∪ {e} does not have a cycle

T := T ∪ {e}

How can we detect cycles? Suppose that there are N nodes V = {n0, n1, ..., nN-1}. We use a variable p that represents a partition of these numbers. That means p is a set of nonempty sets of nodes, each node being in exactly one of these sets, and nothing else is in the sets. We call the members of p its parts. Initially each part of p holds a different node; initially

p = {{n0}, {n1}, ... , {nN-1}}

Together with the initialization of T to the empty set, this establishes a loop invariant that

two nodes u and v are in the same part of p
iff
there is a path from u to v using only edges in T

Assuming this invariant is maintained we have

T {e} has a cycle iff e.source and e.target are in the same part of p

To maintain the invariant, when we add an edge {u,v} to T, we need to merge the parts corresponding to u and v. Define p.part(u) to mean

that part in p that contains u,

p.find(u,v) to mean

p.part(u) = p.part(v),

and p.union(u,v) to mean

p := p \ {p.part(u), p.part(v)} { p.part(u) p.part(v) },

that is p.union(u,v) replaces the parts containing u and v with their union.

We can rewrite the algorithm as

T := ∅

var p := {{n0}, {n1}, ... , {nN-1}}

for each edge e in order of increasing cost

if not p.find( e.source, e.target )

T := T ∪ {e}

p.union(e.source, e.target)

Implementation

Now we come to an implementation in Java.

  • We represent the N nodes by N successive integers starting at 0
  • We represent edges by objects containing the edge's source node, target node and the edge's cost.
  • The graph is just an array of edges.
  • In order to process the edges in increasing order, we put them (actually their indices) into a priority queue and then take them out one at a time. An alternative would be to sort the edge array as a first step; it doesn't really matter.
  • To represent the partition variable p, we use an object that supports the union and find operations mentioned above.

Why it works

First off let's drop the requirement that the graph be connected. A component of an undirected graph is a nonempty subgraph that has, for nodes, a maximal set of nodes that have paths between them in the graph, and for edges all edges between two of its nodes. A forest is the union of a set of pairwise disjoint trees. Equivalently a forest is a set of edges that has no cycles. A spanning forest for a graph is the union of a set of trees for the graph's components. In the special case of a connected graph, a spanning forest and a spanning tree are the same thing. Of course a minimum spanning forest is a spanning forest of least total cost. Any two spanning forests have the same size.

It's clear that, if the graph is connected, the algorithm produces a spanning tree and, in any case, a spanning forest. What may not be clear is that the algorithm produces a minimum spanning tree.

Clearly Kruskal's algorithm is greedy. It makes the locally best choice at each point. It is not at all obvious that this will not lead it down any blind alleys.

The following proof is based on one by Dijkstra.

A simple cycle is a set of 3 or more edges that form a cycle that visits no node more than once. A simple cycle provides two distinct paths between any two nodes on it.

Lemma 0. If a simple cycle C and a spanning forest F share an edge e, we can form a different spanning tree by removing e from F and then adding some other edge of C.

Proof: Removing e = {u,v} from F splits the component of G containing C into two disconnected parts, call these U and V. Some other edge {u',v'} in C must also connect U with V. Edge {u',v'} can not be in F or there would be two paths in F from u to v, one direct and the other through u' and v'. Now F \ {u,v} {u',v'} is a spanning forest. End of Proof.

As two different spanning forests can have the same cost, it is possible that a graph G could have more than one minimum cost spanning forest. However, it simplifies the proof below to consider a total order on the spanning trees. We'll say that e0 < e1 iff cost(e0) < cost(e1) or cost(e0) = cost(e1) and e0 is examined by the algorithm before e1. (Actually, because the algorithm examines edges in order of increasing cost, it would be sufficient to say e0 < e1 iff e0 is examined before e1. For two spanning forests, we'll say that F0 < F1 iff cost(F0) < cost(F1) or cost(F0) = cost(F1) and the least edge in F0 not in F1 < the least edge in F1 not in F0. Now any nonempty set of edges has a unique maximum and there is a unique minimum spanning forest; call it Fmin.

Lemma 1. For any simple cycle C in the graph, the maximum edge in that cycle, {u,v}, can not be Fmin.

Proof: Proof by contradiction. Suppose e were in Fmin. By Lemma 0 we can remove e and replace it with some other edge in the cycle to get another spanning forest. But this edge will be smaller than e. This makes a spanning forest that is smaller than Fmin. End of Proof.

Theorem 2: Every node in Fmin is eventually added to T.

Proof: In each iteration of the loop, an edge e is either added to T or it is rejected. From the way we defined < on edges, all edges in the current value of T are less than e. By Lemma 1, if e is in Fmin, it is not the maximum edge in any simple cycle. As it is larger than every edge in the current value of T, it is therefore not an edge in any cycle involving only itself and the edges in the current value of T. By the description of the algorithm, it is therefore added to T. Since every edge is eventually considered, every edge of Fmin is eventually added to T. End of Proof.

Theorem 3: The final value of T is Fmin.

Proof: From Theorem 2 and the fact that we never remove any nodes from T, it is clear that the final value of T contains all edges of Fmin. It is a loop invariant that the value of T is a forest; so the final value of T is a forest. No strict superset of a spanning forest can also be a forest, so the final value of T can only be Fmin. End of proof.

The invariant trivially holds at the start of the loop. When the loop finishes we have a spanning forest that is a subset of Fmin and so it is Fmin.

Dijkstra's Minimum Spanning Tree Algorithm

Dijkstra came up with an interesting algorithm whose proof is quite similar to the one given above.

T := ∅

For each edge e in any order at all

Add e to T

If this results in a cycle, remove the largest edge in that cycle.

The really interesting thing about this algorithm is that you can examine the edges in any order you like. Depending on the order you get different algorithms, but all these algorithms share a single proof.

If you examine the edges in increasing cost order, you have Kruskal's algorithm.

You get the Jarník/Prim algorithm if the edges are examined in an order such that the first edge is a least cost edge and, after that, the next edge to be examined is a least cost edge among those that share a vertex with an edge already in T.

You get Borůvka's algorithm if you examine the edges in yet a different order. The list of edges consists of a number of rounds. The first round of edges examined are the smallest ones that group the vertices into pairs. That is, the first round contains the smallest edge and the smallest edge that does not meet the smallest edge and so on. (Within this round, and all others, the edges can be examined in any order.) Now the vertices are partitioned into a number of parts. Most parts are pairs, but there may be a left-over singleton part. The second round of edges are the smallest ones that join pairs of these parts. And so on, until no more pairs of parts can be joined. At this point, all remaining edges would form cycles and are larger than the ones in the forest and so may be ignored.
